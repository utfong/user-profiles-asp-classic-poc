# CLAUDE.md: asp-jscript

Guidance for working in this specific host. See [`../CLAUDE.md`](../CLAUDE.md) first for repo-wide IIS/Access environment gotchas that apply here too. See [`README.md`](README.md) for the user-facing overview, API reference, and setup steps.

## Architecture in one paragraph

Same shape as `asp-vbscript`: static HTML/CSS/axios frontend (`index.html`, `assets/` - copied verbatim, since the frontend has zero backend-language dependency) talks to a pure-JSON ASP API (`api/session.asp`, `api/profiles.asp`). No database ships with the app: `api/session.asp` provisions a session-scoped `.accdb` under `App_Data/sessions/{SessionID}.accdb` on demand, and `global.asa`'s `Session_OnEnd` (plus an explicit `DELETE /api/session.asp`) tears it back down. Writes require an active session + CSRF token; reads don't. This host was built as a genuine JScript port, not a syntax-translated copy of `asp-vbscript` - see "What differs from a mechanical port" below for where the two hosts' internals actually diverge.

## Conventions to preserve

- **camelCase for everything authored here** (`getRequestMethod`, `hasActiveSession`, `sendData`, `jsonParse`), native PascalCase left alone for COM surface not owned by this project (`Response.Write`, `Server.CreateObject`, `cmd.CreateParameter`). No `Option Explicit` equivalent exists in JScript - there's no enforced-declaration pragma, `var`-declare everything as a discipline.
- **Real `try/catch/finally`** everywhere VBScript's sibling used `On Error Resume Next`/`Err.Number`/`On Error Goto 0`. Most visible in `executeLocked` (`api/profiles.asp`): `finally { Application.Unlock(); }` expresses "guaranteed unlock" directly rather than approximating it with manual Err-state bookkeeping.
- **`Response.End()` aborts the entire request**, confirmed to behave the same as VBScript's `Response.End` (unwinds past any surrounding `try/catch`, uncaught) via a throwaway probe before this host was built. `sendData`/`sendError`/`sendJson` in `lib/http.asp` rely on this; call sites never need an explicit `return` after them to prevent fallthrough.
- **`CreateParameter`'s `Size` argument must be a real value, never omitted.** VBScript's `cmd.CreateParameter(name, type, dir, , value)` skipped-argument comma syntax doesn't exist in JScript - passing nothing there is a syntax error. Pass an explicit placeholder (`0`) for types where `Size` is unused (e.g. `adDate`).
- **A JScript `Date` passed as an ADO parameter value needs `.getVarDate()` first.** `cmd.CreateParameter("CreatedAt", 7, 1, 0, new Date())` throws "Application uses a value of the wrong type for the current operation"; `(new Date()).getVarDate()` is the correct COM-compatible conversion (confirmed via probe).
- **ADO `DATETIME` fields read back from a `Recordset` are a raw COM `VarDate`, not a JScript `Date`.** `typeof` reports `"date"`, `instanceof Date` is `false`, and the value has no methods of its own (`"type 'VarDate' does not support user-defined properties"`). `rowToObject` (`api/profiles.asp`) normalizes this with `new Date(rs("CreatedAt").Value)` at the point of reading it out of the recordset, so `jsonEncode` only ever needs to handle genuine `Date` instances via `instanceof Date`. (`lib/json.asp`'s `jsonEncode` also has a defensive `typeof value === "date"` fallback branch in case a raw `VarDate` ever slips through unnormalized.)
- **`Request.QueryString(key)`/`Request.ServerVariables(key)` return a COM collection object even when `key` doesn't exist in the request.** Stringifying that empty collection produces the literal 9-character text `"undefined"`, not an empty string or JScript's own `undefined` primitive - so `(x === undefined) ? "" : String(x)` does **not** correctly detect "absent." This caused a real bug during development: `GET /api/profiles.asp` with no `id` at all was routing into `handleGet("undefined")` instead of `handleList()`. Fixed with `requestParam(collectionItem)` in `lib/http.asp`, which checks `.Count === 0` instead - use it for any query-string/server-variable read that might legitimately be absent (mandatory fields like `REQUEST_METHOD` or `HTTP_HOST` don't need it, since they're never actually missing from a real request).
- **JSON is hand-rolled** (`lib/json.asp`, since `typeof JSON === "undefined"` in this engine - confirmed empirically, no native JSON global despite `@_jscript_version` reporting 11) but, unlike `asp-vbscript`'s flat-objects-only parser, **`jsonParse` here supports full nesting** (objects/arrays of arbitrary depth). VBScript's parser was flat-only because rejecting nesting was *easier* than supporting it (no natural place to put a nested value without extra `Scripting.Dictionary` bookkeeping); in JScript, rejecting nesting would take *more* code (an explicit throw-on-`{`/`[` branch) than just recursing into it via native object/array literals, so there's no simplicity win in artificially limiting it. Parser state (`pos`/`len`) lives in closures local to each `jsonParse()` call rather than a `Class` the way VBScript's `ClsFlatJsonParser` needed - JScript's function-scoped closures make the object unnecessary.
- **Response envelope**: every API response is `{"data": ...}` or `{"error": {"code": "...", "message": "..."}}`, via `sendData`/`sendError` in `lib/http.asp`. Keep new endpoints consistent with this - the frontend (shared verbatim with `asp-vbscript`) depends on this contract matching exactly.

## Security requirements for any new endpoint

- Gate every write (`POST`/`PUT`/`DELETE`) behind `requireSession()` + `requireCsrf()` (see `lib/db.asp`, `lib/csrf.asp`). Reads can skip both.
- Use parameterized `ADODB.Command` objects: never concatenate user input into SQL text.
- Validate required fields and max lengths server-side before hitting the DB (see `validateProfileFields` in `api/profiles.asp`) - don't rely on the DB column size or the frontend's `required`/`maxlength` attributes alone. Return a `400 validation_error` with a combined message, not a raw ADO error.
- Wrap writes in `Application.Lock()`/`Unlock()` (use the existing `executeLocked` helper in `api/profiles.asp` rather than inlining a new Lock/Unlock pair) so a failed write can't hang the app pool. Use `finally` for the unlock, not a duplicated unlock call on every exit path.
- HTML-encode anything rendered into the DOM from user-supplied data: `assets/app.js` (shared with `asp-vbscript`) builds rows with `textContent`/DOM APIs rather than string-concatenated `innerHTML` for exactly this reason; keep doing that.
- Any new `Request.QueryString`/`Request.ServerVariables` read for a value that might legitimately be absent needs `requestParam()` (see above) - don't reintroduce the `undefined`-string bug.

## Frontend logic split: app.core.js vs app.js

Identical structure and rationale to `asp-vbscript` (the files are copied verbatim - see README). `assets/app.core.js` holds pure(ish) logic (`extractErrorMessage`, `textCell`, `buildProfilePayload`, `createConfirmController`) exposed as `window.App`; `assets/app.js` does DOM lookups, axios calls, and event wiring, calling into `App.*` rather than reimplementing logic locally. When adding new frontend logic that doesn't need `api`/live DOM event wiring, put it in `app.core.js`, not `app.js`.

## Testing

`tests/run.asp` is the whole story for the backend; see the README's Testing section. Same suite shape as `asp-vbscript` (`json`, `csrf`, `db`, `api`), but two differences:

- **`tests/api.tests.asp` uses `WinHttp.WinHttpRequest.5.1`, not `MSXML2.ServerXMLHTTP.6.0`.** `asp-vbscript`'s suite uses the latter successfully from VBScript, but the identical component's `.Open()` method is not resolvable through JScript's `IDispatch` calls on this machine - confirmed via a throwaway probe (`"Open" in obj` is `false`, calling it throws `"Object doesn't support property or method 'Open'"`, and this reproduces for both `MSXML2.ServerXMLHTTP.6.0` and the version-independent `MSXML2.ServerXMLHTTP` ProgID). `WinHttpRequest` exposes the same `Open`/`setRequestHeader`/`send`/`status`/`responseText`/`getResponseHeader` surface and works correctly from JScript here. If this component ever needs swapping again, `Msxml2.XMLHTTP.6.0` (the client-side, non-Server variant) also has a working `Open` method on this machine, as a second fallback.
- **`tests/api.tests.asp` calls `jsonParse()` directly on live HTTP response bodies** (including the `{"data": ...}`/`{"error": {...}}` envelope) instead of `asp-vbscript`'s `InStr`/`Extract*` string-scraping helpers, since nesting is supported here.

For the frontend, `assets/tests.html` (+ `assets/testing.js`, `assets/tests.core.js`) is copied verbatim from `asp-vbscript` and needs no adaptation - it only exercises `app.core.js`, which has no backend dependency.

`a11y.html` (+ `assets/a11y.run.js`) is also copied verbatim - it fetches whatever `index.html` currently contains at request time, so it needs no host-specific changes either.

## The intermittent script-engine crash - what was investigated, what worked

**Status: resolved.** `jscript-engine-fix.ps1 enable` (repo root) fixes it - see below for what that actually does and what didn't work first.

**Symptom:** a meaningful fraction of requests to any `.asp` page here (roughly half, empirically, before the fix) failed with `500` / `ASP 0240 Script Engine Exception`: `A ScriptEngine threw exception 'C0000005' in 'IActiveScript::SetScriptState()' from 'CActiveScriptEngine::ReuseEngine()'`. Reproduced on `index.asp` (zero ADO/COM usage, since deleted) just as readily as on the heavier `api/profiles.asp`, and never happened on `asp-vbscript` (same machine, same IIS).

**Root cause:** Windows 11 24H2/25H2 (this machine: 25H2, build 26200) defaults to a newer `jscript9legacy.dll` script engine for security hardening, which isn't fully compatible with classic ASP's script-engine-reuse-across-requests model.

**Three registry-based fixes were tried, in this order, via `jscript-engine-fix.ps1` (originally a `.cmd`, later converted to `.ps1` so the script can't be triggered by an accidental Explorer double-click - see its header comment; each fix was enabled → retested in isolation → disabled if it didn't help):**

1. ~~`FEATURE_ENABLE_PERSISTENCE`~~ - `HKLM\Software\Policies\Microsoft\Internet Explorer\Main\FeatureControl\FEATURE_ENABLE_PERSISTENCE`, DWORD `w3wp.exe` = `1`. Microsoft's official fix (KB5077241+, Feb 2026; this machine has a later cumulative update, KB5101650 July 2026) for JScript globals *not persisting* across script executions (a silent-data-loss bug) - turned out to be a different defect from this crash, even though both plausibly stem from the same `jscript9legacy.dll` swap. **Did not help** (crash rate unchanged, ~2-3/10 success on small samples, identical event log signature). Removed from the script after the third fix (below) was confirmed working - no reason to keep dead options as live toggles.
2. ~~`FEATURE_ENABLE_JSCRIPT9_LEGACY`~~ - `HKLM\SOFTWARE\Microsoft\Internet Explorer\Main\FeatureControl\FEATURE_ENABLE_JSCRIPT9_LEGACY`, DWORD `w3wp.exe` = `0` (+ WOW6432Node mirror). A per-process opt-out of `jscript9legacy.dll` entirely, reverting to the older `jscript.dll` (5.8) for that process. Lower-confidence source than #1 (a Microsoft Windows IT Pro Blog post that couldn't be fetched directly to verify, only a search-engine summary). **Did not help either** (same pattern as #1). Also removed from the script.
3. **`JScriptReplacement` - the one that worked.** `HKLM\SOFTWARE\Policies\Microsoft\Internet Explorer\Main`, DWORD value `JScriptReplacement` = `0`. Unlike #1/#2, this is **not per-process** - a single machine-wide value affecting every process using JScript on the machine, not just `w3wp.exe`. Source: found via a general web search early in the investigation, deprioritized at the time in favor of #1's more rigorously-verified-at-the-time sourcing, circled back to once #1 and #2 both failed. Confirmed via 40+ requests across `smoketest.asp`, `api/profiles.asp`, and a full `tests/run.asp` run: 0 failures, clean event log, versus the ~50% baseline failure rate beforehand. This is what `jscript-engine-fix.ps1 enable`/`disable`/`status` operates on now.

**Why this fix and not #1/#2:** genuinely unclear - all three are plausibly related to the same underlying `jscript9legacy.dll` engine swap, but only the machine-wide DLL-selection override (#3) actually changed the crash rate; the two per-process behavior/engine-selection flags (#1, #2) did not, despite #2 nominally claiming to do something similar (opt a process out of the new engine). Possible explanation: `JScriptReplacement` may act earlier/more fundamentally in the engine-loading path than the per-process `FeatureControl` flags do, but this hasn't been verified beyond the observed effect.

**What was tried and ruled out, so it isn't repeated:**
- `<system.webServer><asp><cache scriptEngineCacheMax="0" /></asp></system.webServer>` in `web.config`, to disable IIS's script-engine pooling entirely (a standard fix for engine-reuse instability, in principle - the crash literally names `ReuseEngine` in its stack). **Do not do this on this machine.** This machine's classic ASP feature has no registered config schema for it (`%windir%\System32\inetsrv\config\schema\asp_schema.xml` doesn't exist here - only `ASPNET_schema.xml`, for ASP.NET, a different feature). The `<asp>` element isn't just ineffective without that schema, it breaks the site's entire `web.config` parse, which manifests as every request failing with "New Application Failed" and can trip IIS's Rapid-Fail Protection, stopping the app pool outright (happened twice during development; recovered both times, once via IIS's own timeout-based rapid-fail cooldown and once after removing the `<asp>` block).
- A one-off `regsvr32 jscript.dll` re-registration was considered but not applied - a read-only registry check first showed `jscript.dll` (not `jscript9legacy.dll`) is already the actively-registered COM engine for the `JScript` ProgID on this machine (`HKLM:\SOFTWARE\Classes\JScript\CLSID` → `HKLM:\SOFTWARE\Classes\CLSID\{that CLSID}\InprocServer32`), so re-registering it would be a no-op - the regression isn't about the wrong DLL being *registered*, it's `jscript.dll`'s own current build behaving like the newer engine after a recent Windows Update (the file's own last-write timestamp was one day before this host was built).
- `iisreset` (used by the first version of `jscript-engine-fix.ps1`) failed mid-restart with "Access denied, you must be an administrator of the remote computer to use this command" even from a genuinely elevated prompt, leaving `W3SVC`/`WAS` fully stopped with no automatic recovery (both hosts down until manually recovered via `net start w3svc`). The script now uses `net stop`/`net start w3svc` instead, and explicitly verifies the service came back up rather than assuming success - if you need to restart IIS manually for any reason, prefer `net stop`/`net start w3svc` over `iisreset` on this machine.

**If working on a fresh clone/setup and `jscript-engine-fix.ps1 status` shows the fix not enabled:** treat any `500` as "retry the request" first, not "there's a bug" - it's this same crash. A crash takes down the whole worker process (not just one script engine instance), which wipes *all* in-memory session state for *all* active sessions - if a request immediately following a crash unexpectedly reports `no_session`, that's why, not a session-handling bug in this code. `tests/run.asp` chains roughly a dozen sequential self-HTTP-calls in one execution, so without the fix a full clean run needs a few attempts (the crash probability compounds across the chain); `tests/api.tests.asp`'s `httpCall()` also has its own internal retry logic (up to 4 attempts per self-call, see that file) as a second layer of defense independent of whether the registry fix is applied. Most of the application logic was independently verified correct via isolated single requests during the original development/debugging process (list, create, and the `requestParam` fix were each confirmed correct on their own before the fix existed), so pre-fix flakiness was never evidence of an actual application bug.
